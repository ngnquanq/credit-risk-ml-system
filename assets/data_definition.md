# Home Credit Default Risk – Input/Derived Features and Data Flow

## Data flow for credit‑risk modelling

When a customer applies for a Home Credit loan, their information enters the platform via an **online transactional processing (OLTP) system**.  The OLTP database stores transactional information (products, transactions, employees, customers and suppliers) and supports rapid inserts and updates:contentReference[oaicite:0]{index=0}.  An **extract, transform and load (ETL)** process then pulls data from the OLTP database into a staging area where it is cleaned and transformed, and finally loads it into an **online analytical processing (OLAP) database**, i.e., the data warehouse:contentReference[oaicite:1]{index=1}.  A **data mart** acts as an access layer of the data warehouse, providing summarized data for specific analytical tasks:contentReference[oaicite:2]{index=2}.  This pipeline enables credit‑risk models to use both transactional data and enriched features.

1. **User sign‑up and data capture (OLTP).**  When the customer applies for a loan, they supply personal, financial and employment information.  The system also collects metadata such as time of application and whether certain documents or contact details were provided.  These fields are stored in the `application_train/application_test` table.  Other tables (bureau, previous_application, POS_CASH_balance, credit_card_balance and installments_payments) are populated by asynchronous processes querying credit bureaus or internal systems and therefore do not require new user input.

2. **Integration and feature engineering (ETL to OLAP).**  The ETL process pulls the raw application and external data into a staging area, where it is cleansed and **derived features** are computed.  Examples include external credit scores (`EXT_SOURCE_1–3`), region population metrics, normalized building statistics and ratios of different amounts.  The transformed data is loaded into the data warehouse and aggregated at the loan (`SK_ID_CURR`) level.  

3. **Data mart and model serving.**  A data mart provides only the variables needed for model serving.  A REST API (for example, built with FastAPI) would accept **user‑supplied features** from the current application and join them with **pre‑computed features** from the data mart.  Derived variables are calculated (e.g., employment duration, age) and the resulting feature vector is passed to the credit‑risk model to produce a probability of default.  

The table below summarises which features require user input and which are derived from external data or calculated during ETL.

## Mapping of data tables and feature types

| Table | Features taken directly from user input or easily derived | Derived/external/calculated features |
| --- | --- | --- |
| **application\\_{train\\|test}.csv** | **Customer‑supplied demographics, employment and loan request data:** contract type, gender, car/house ownership flags, number of children, total income, requested credit amount, annuity, goods price, who accompanied applicant, income type, education level, family status, housing type and occupation.  Personal attributes like age (derived from `DAYS_BIRTH`), employment length (`DAYS_EMPLOYED`), registration/ID change dates (`DAYS_REGISTRATION`, `DAYS_ID_PUBLISH`) and car age are also computed from user‑provided dates.  Flags indicating whether the applicant provided mobile phone, work phone, home phone or email are captured directly.  Counts of family members and children and the number of documents provided (FLAG_DOCUMENT_2–FLAG_DOCUMENT_21) are also user supplied. | **System/external or engineered variables:** region population and ratings (`REGION_POPULATION_RELATIVE`, `REGION_RATING_CLIENT`, `REGION_RATING_CLIENT_W_CITY`) reflect the applicant’s location and are computed by the system.  Flags that compare permanent, contact and work addresses (`REG_REGION_NOT_LIVE_REGION`, `LIVE_REGION_NOT_WORK_REGION`, etc.) are derived comparisons.  External credit‑score features (`EXT_SOURCE_1`, `EXT_SOURCE_2`, `EXT_SOURCE_3`) come from third‑party bureaus.  Building and property attributes such as `APARTMENTS_AVG`, `BASEMENTAREA_AVG`, `ELEVATORS_AVG`, `_MODE`, `_MEDI` etc. provide normalized information about the applicant’s residence and are derived from real‑estate databases.  Social‑circle statistics (e.g., number of contacts who defaulted) and credit bureau inquiry counts (`AMT_REQ_CREDIT_BUREAU_*`) are also external/derived. |
| **bureau.csv** | None – the table contains information about the applicant’s previous loans from other institutions. | **External credit bureau data:** includes credit status (`CREDIT_ACTIVE`), currency, time since credit was opened or ended (`DAYS_CREDIT`, `DAYS_CREDIT_ENDDATE`, `DAYS_ENDDATE_FACT`), overdue days, maximum overdue amount, credit amounts (`AMT_CREDIT_SUM`, debt, limit, overdue), number of prolongations and type of credit.  All of these are gathered from credit bureaux and not provided by the applicant. |
| **bureau_balance.csv** | None. | **Monthly status of each bureau credit:** includes month index relative to application (`MONTHS_BALANCE`) and status codes indicating whether the credit is active, closed or in different delinquency buckets.  These values are generated by the bureau. |
| **previous_application.csv** | The applicant’s earlier loan applications within Home Credit – although originally user‑supplied when those loans were applied for, they are historical and not re‑entered. | **Historical application data:** contract type, annuity, amount requested, amount granted, down payment, goods price, application day of week and hour, flags for last application in contract/day, micro‑loan flag, down payment rate, interest rates, loan purpose, status and decision dates etc. are stored for each prior application.  Many fields are normalized (e.g., `RATE_DOWN_PAYMENT`), so they are considered derived. |
| **POS_CASH_balance.csv** | None. | **Point‑of‑sale/cash loan history:** For each previous loan, this table records monthly balance information such as months balance, number of installments left, contract status and delinquency days (`SK_DPD`, `SK_DPD_DEF`).  These are derived from the bank’s records. |
| **credit_card_balance.csv** | None. | **Credit‑card usage history:** monthly balances, credit limits, drawing amounts (ATM, purchases, other), minimum installment amounts, total payments, receivables, and counts of drawings or installments are recorded for each past credit card.  These features are generated from transaction records and are not provided by the user. |
| **installments_payments.csv** | None. | **Payment history:** for each previous loan, this table lists the installment schedule (`NUM_INSTALMENT_VERSION`, `NUM_INSTALMENT_NUMBER`), due date (`DAYS_INSTALMENT`), actual payment date (`DAYS_ENTRY_PAYMENT`), prescribed installment amount and the amount actually paid.  These values come from the lending system’s records and not from current user input. |
| **Other files** (e.g., `sample_submission.csv`) | Not applicable. | These files provide examples of submission format and do not contain features. |

## Implications for building a FastAPI endpoint

* **User‑input payload:** The API should accept only the fields that the user can realistically provide during the application process (highlighted in the left column above).  Many variables (e.g., age, employment duration, family size, occupation, housing type, requested loan amount, income and whether they own a car/house) can be collected directly from the user.  Relative time fields (`DAYS_BIRTH`, `DAYS_EMPLOYED`) can be derived from dates supplied by the user.  Flags indicating whether documents or contact details are provided can be captured through simple check boxes.
* **Derived/external features:** Features from credit bureaux, previous applications and financial transaction tables should **not** be requested from the user.  These will be populated by back‑office processes.  Normalized regional and building attributes, social‑circle statistics and external scores (EXT_SOURCE_1–3) should be computed in the data warehouse and joined by the ETL process.
* **Data flow clarity:** The OLTP system gathers user inputs and obtains external data asynchronously.  ETL processes transfer this data to the OLAP warehouse where feature engineering occurs:contentReference[oaicite:22]{index=22}.  A data mart exposes the engineered features to the model‑serving layer, and the FastAPI endpoint combines real‑time user inputs with pre‑computed features to generate a probability of default.

This mapping ensures that only appropriate inputs are solicited from applicants while leveraging rich external and engineered data for accurate credit‑risk assessment.
